# Growth / Conversion / SEO / Shareability / Robustness Audit

**Site:** Land Selection Framework — https://land-selection-framework.vercel.app
**Lens:** Opportunity cost during a high-traffic window — "what are we failing to capture, and what breaks under load?"
**Audited:** 2026-05-25
**Scope note:** Findings are against local files in `prototype/`. Live deploy was spot-checked (OG images, PDF, robots/sitemap, canonical, stale copy) and matches local where checked, but **local files may be ahead of or behind the live deploy** — re-verify before shipping fixes.
**Out of scope (handled separately):** stale "four regions" copy in hero/lede/key/regions-intro heading, stale footer dates.

## CRITICAL

### C1 — The modal gates the core interaction behind an email ask, and fires on a timer with no engagement signal
**Location:** `src/main.js` lines 864–906; CSS `body.interaction-locked` in `index.html` lines 819–828.

**What it does (quoting the trigger logic):**
```js
const LOAD_DELAY_MS = 3000;        // pop the modal this long after page load
const SLIDER_LOCK_GRACE_MS = 600;  // brief grace at load before sliders fade
...
// Lock sliders softly until interaction
setTimeout(() => {
  if (!modalShown) {
    document.body.classList.add('interaction-locked');
  }
}, SLIDER_LOCK_GRACE_MS);

// Fire the modal 1.5s after page load — no scroll requirement
setTimeout(() => {
  if (!modalShown) showModal();
}, LOAD_DELAY_MS);
```
And the CSS gate:
```css
body.interaction-locked .criteria .slider-block,
body.interaction-locked .criteria .reset {
  pointer-events: none;
  opacity: 0.42;
  filter: grayscale(0.3);
}
```

**What's wrong:**
1. **Trigger is a pure load timer, not engagement.** Despite the section header ("3s after first scroll") and the inline comment ("1.5s after page load"), the actual behavior is: `interaction-locked` added at **600ms**, modal shown at **3000ms** — regardless of scroll, mouse movement, or any interest signal. The two comments are both stale and contradict the code. A cold visitor who has done nothing gets interrupted in 3 seconds.
2. **The sliders — the entire reason the page is interesting — are disabled at 600ms** and stay disabled until the modal is dismissed or submitted (`dismissModal` removes `interaction-locked`, lines 927). So the value-demonstration mechanism is greyed out (`opacity: 0.42; grayscale`) *before* the visitor has seen it work, and the email modal is the thing standing between them and it. This is a hard interaction gate, not a soft nudge.
3. The lock only releases on modal dismiss/submit — there is no path to the sliders that doesn't route through the email prompt on first visit.

**Why it matters under high traffic:** A cold, high-volume audience (e.g. arriving from a single shared link or social post) has the lowest intent and the highest bounce sensitivity. Interrupting at 3s with a modal *and* freezing the headline interaction is close to a worst-case first impression: it reads as "demo a feature, then paywall it with your email." The likely outcome is elevated bounce and *fewer* signups than a softer ask, because the people who would have subscribed are the ones who first got to play with the sliders and found them valuable — and you have disabled exactly that.

**Suggested fix (in priority order):**
- **Stop locking the sliders entirely.** Remove the `interaction-locked` add (line 897–901) and ideally the CSS block (819–828). Let the modal float over a live page. This is the single highest-leverage change.
- **Delay and gate the modal on a real signal.** Fire on scroll-depth (e.g. user reaches the criteria section) OR a longer dwell (20–30s), whichever first; or exit-intent on desktop. Keep `?modal=1` test override.
- **Keep the localStorage suppression** (`lsf-modal-state`) — that part is correct and prevents re-nagging.
- **Effort: S** (delete lock + change one setTimeout to a scroll/IntersectionObserver trigger).

### C2 — No canonical URL, no robots.txt, no sitemap.xml
**Location:** `index.html` `<head>` (no `<link rel="canonical">`); live `/robots.txt` → 404; live `/sitemap.xml` → 404 (verified on deploy).

**What's wrong:** There is no canonical tag on either page. Under high traffic the site will be shared with query strings appended — most notably the framework's *own* `?t.climate=…` share URLs (see S1) and any `?modal=1`, plus the usual `?utm_*`, `?fbclid`, `?ref=` from social platforms. Each variant is a distinct URL to a crawler, splitting any ranking signal and risking duplicate-content dilution exactly when inbound links are spiking. No sitemap/robots means `deeper.html` (the richest, most search-valuable page) may be discovered slowly or not at all.

**Why it matters under high traffic:** A traffic spike is when backlinks and shares accumulate — the moment canonicalization pays off most. Getting this wrong now wastes the SEO equity the spike generates. `deeper.html` is genuinely strong long-form content (methodology + three case studies + huge sourced bibliography) and is the asset most likely to rank for "where to start an ecovillage / regenerative community land" — but it is currently under-surfaced.

**Suggested fix:**
- Add `<link rel="canonical" href="https://land-selection-framework.vercel.app/" />` to `index.html` and the self-URL to `deeper.html`.
- Add `robots.txt` (allow all + sitemap line) and a 4-line `sitemap.xml` listing `/` and `/deeper.html`. Both are static files; Vercel serves them from the prototype root.
- **Effort: S**

## HIGH

### H1 — Triple email-capture surface with no differentiation creates redundancy and "asked 3 times" fatigue
**Location:** hero signup card (`index.html` 1146–1165), modal (1328–1360), contact-section form is *implied by CSS* (`.contact form` 703–735) but **note:** the contact section in the current HTML (1227–1280) renders social links + a mailto invitation, not an email-capture `<form>`. So there are effectively **two** active Web3Forms capture forms (hero + modal) plus a styled-but-unused contact-form CSS block.

**What's wrong:**
- The hero card and the modal make **the same ask with overlapping copy** ("Stay in touch" / "Want to keep up with this work?") and post to the same Web3Forms key with only the `subject` field distinguishing them ("hero" vs "modal signup"). A visitor who ignored the hero card gets the modal 3s later asking again. Redundancy without a reason.
- The value exchange is vague: "Low-frequency updates as this work evolves" (hero) and "additional regions, deeper data layers" (modal). Neither names a concrete, immediate payoff. The modal *does* gesture at "Subscribers see it first" but the deeper material is already public at `/deeper.html` with no gate — so the implied exclusivity isn't real.
- There is dead CSS (`.contact form`, `.contact input[type="email"]`, `.contact button`, `.contact .form-status`, lines 703–735) for a contact form that no longer exists in markup. Cleanup item.

**Why it matters under high traffic:** When many cold visitors hit at once, a clear single value-exchange converts better than two diffuse ones. The current setup spends two interruptions on the same undifferentiated ask.

**Suggested fix:**
- Pick **one** primary surface. Recommended: keep the hero card as the always-visible ask; convert the modal to a *single, later, engagement-triggered* reinforcement (see C1) — or drop the modal and rely on hero + an inline CTA after the criteria grid.
- Make the value-exchange concrete and honest: e.g. "When the next regions land (Cévennes, Estonia, Saxony-Anhalt) and when the working-group invitation opens, I'll email you — roughly monthly." That ties the ask to the "What's coming next" content that already exists in `deeper.html`.
- Remove dead `.contact form` CSS.
- **Effort: M**

### H2 — "v1 sketch" self-deprecation is everywhere and undercuts the ask
**Location:** `<title>` "— v1 sketch" (index 6); OG/Twitter titles (12, 18); hero eyebrow "v1 sketch · May 2026" (1129); footer stamp "v1 sketch" (1319); deeper footer (875). Also hero lede "This site sketches…" and "the framework is a sketch, not a product."

**What's wrong:** The word "sketch" is the single most repeated framing token on the site — it's in the browser tab, the social share card title, the hero eyebrow, and both footers. The intent (honesty, not overselling) is sound and aligns with the project's tier-honesty discipline. But for a *high-traffic, first-impression* audience it does three things: (1) it tells search engines and social-card readers the canonical name of the thing is "Land Selection Framework — v1 sketch", weakening the brandable title; (2) it pre-emptively discounts the work before the visitor has judged it; (3) it weakens the email ask and working-group invitation — "why subscribe to / join a sketch?"

**Why it matters under high traffic:** First-time visitors at scale calibrate trust fast. The data layer here is genuinely substantial (sourced, vintaged, 10 regions, a real bibliography). Leading with "sketch" in the tab and share card sells it short at exactly the moment credibility converts to signups.

**Suggested fix:** Keep *one* honest "v1 / in-progress" signal in-body (e.g. the methodology "What V1 cannot do yet" section is the right place — it's honest in context). Remove "sketch" from `<title>`, OG title, and Twitter title — make those just "Land Selection Framework — siting regenerative settlements across Europe" or similar keyword-bearing phrasing. This is a framing/preference call for Gustaf, not a factual fix.
- **Effort: S**

### H3 — Authorship vs working-group relationship is explained, but buried in the footer
**Location:** footer "About this sketch" (1287–1302); contact invitation-cta (1271–1276).

**What's wrong:** The credibility story — Gustaf built the public sketch, the framework is a private working group's work, attribution pending — is actually *well written* and honest. But it lives only in the footer, which most of a bounce-prone high-traffic audience never reaches. A first-time visitor sees an anonymous-feeling dashboard asking for their email with no visible "who is behind this and why should I trust them" until they scroll to the very bottom.

**Why it matters under high traffic:** Trust signals need to be near the ask. The hero signup card says "Your address goes only to me" — but "me" is undefined at that point on the page.

**Suggested fix:** Add a one-line authorship/credibility signal near the hero or signup card (e.g. "Built by Gustaf Palm as a public sketch of a broader working-group project" with a link to the footer/about or deeper page). Low effort, meaningfully raises trust at the point of conversion.
- **Effort: S**

### H4 — Per-region OG variant logic can silently point share cards at a 404-or-stale image
**Location:** `src/main.js` 150–172 (`REGIONS_WITH_OG_VARIANT`, `pickOgImagePath`, `updateOgImageMeta`).

**What's wrong:** The dynamic og:image swap is clever but has two issues:
1. **It mutates `og:image` client-side via JS.** Social scrapers (Facebook, LinkedIn, X, WhatsApp, Slack) read the *server-returned static HTML* and do **not** execute the module script. So the dynamic swap to `og-alentejo.png` / `og-filtered.png` based on filter state **never reaches a share crawler**. Every shared link — regardless of filter state — will use the static `og.png` from the HTML source. The entire `pickOgImagePath` mechanism is effectively dead for its intended purpose (it only changes the live DOM, which no scraper sees).
2. The variant set is `alentejo, galicia, transylvania, connemara` — only 4 of 10 regions have a PNG. The code correctly falls back to `og-filtered.png` for the other 6, so no 404 — but combined with (1), this whole branch is moot for sharing.

**Verified:** `og.png`, `og-alentejo.png`, `og-filtered.png` all return **200** on the live deploy, and `public/` (the og-*.html generators) is correctly excluded via `.vercelignore`. So the *images exist*; they just can't be selected dynamically by crawlers.

**Why it matters under high traffic:** This is the viral-loop image. If someone filters to "only Alentejo passes" and shares, the intent was a region-specific card — but they'll get the generic `og.png`. Not broken, just a feature that doesn't do what it looks like it does. Worth knowing before relying on it.

**Suggested fix:** Either (a) accept that all shares use `og.png` and delete the dynamic-swap code as dead weight, or (b) if region-specific cards matter, generate them at share time server-side (not feasible on a static Vercel site without a function) — so realistically (a). At minimum, make `og.png` itself excellent since it's the only card that ships. **Effort: S** (delete) — or document that it's intentionally cosmetic for the live DOM only.

## MEDIUM

### M1 — Single points of failure on third-party runtime dependencies
**Location:** `index.html` 24–29 (Google Fonts, MapLibre from unpkg); `src/main.js` map tile sources (184–364).

**External runtime dependencies and failure modes under a spike:**
| Dependency | Source | If it rate-limits / fails | Visible failure |
|---|---|---|---|
| MapLibre GL JS + CSS | `unpkg.com` | unpkg has had outages; CDN throttling | `maplibregl is undefined` → `initMap()` throws → **map never renders**, but rest of page (criteria, sliders, table) still works because `initMap()` is called last and the throw is uncaught after the renders. Console error; empty grey map box. |
| Google Fonts (Inter, Spectral) | `fonts.googleapis.com` | slow/blocked (e.g. some regions/networks) | FOUT/FOIT — falls back to Georgia/system-ui. Cosmetic. |
| CARTO basemap tiles | `*.basemaps.cartocdn.com` | rate-limit on tile bursts | grey/blank base tiles |
| Global Forest Watch tiles | `tiles.globalforestwatch.org` | GFW is frequently slow/flaky | layer just doesn't paint (off by default — low impact) |
| Esri/OpenTopoMap/EOX/NASA GIBS | various | OpenTopoMap especially rate-limits aggressively | layer blank when toggled on (all off by default except ecovillages) |
| Aqueduct/UCDP/ecovillage GeoJSON | `data/processed/*` | **self-hosted on Vercel** — safe | n/a (good — these are local) |

**Assessment:** The map data overlays are mostly off-by-default and self-hosted GeoJSON, so the worst realistic failure is the **base map + MapLibre lib**. MapLibre from unpkg is the most consequential SPOF: if unpkg throttles during a spike, the map silently fails to initialize. The page degrades gracefully-ish (the map is not the core interaction — the sliders are) but it looks broken.

**Suggested fixes:**
- **Pin + self-host MapLibre** (download the 4.7.1 dist into the repo and serve from Vercel) — removes the unpkg SPOF and is the highest-value mitigation. **Effort: S.**
- Self-host the two Google fonts (or accept the system fallback — it's fine). **Effort: S–M.**
- Wrap `initMap()` in try/catch and hide/placeholder the map box on failure so a dead map reads as "map unavailable" rather than a grey void. **Effort: S.**
- Lazy-init the map via IntersectionObserver so tile requests only fire when the map scrolls into view — reduces tile-burst load and speeds first paint. **Effort: M.**

### M2 — Signup conversion is NOT measurable; only pageviews are
**Location:** `index.html` 31–32 (Vercel Analytics script); form handlers in `main.js` (820–861, 964–996).

**What's wrong:** Vercel Analytics (Hobby, script-injected) tracks pageviews and basic web vitals only. The signup success paths (`setStatus(status, 'success', …)`) fire **no analytics event**. So during the high-traffic window you will see *traffic* but have **no idea what your signup conversion rate is**, which surface (hero vs modal) converts, or whether the modal helps or hurts. You're flying blind on the one metric that matters for the opportunity.

**Why it matters under high traffic:** This is the week you'd most want to A/B or even just *measure* the funnel. Without an event on submit-success you can't tell if C1/H1 fixes helped.

**Suggested fix:** Vercel Analytics supports custom events via `window.va?.('event', { name: 'signup', surface: 'hero'|'modal' })`. Add one line in each success branch and one for modal-dismissed. Also track `share_clicked`. **Effort: S.**

### M3 — Share round-trip works; share *discoverability* and previews are weak
**Location:** `main.js` URL state (102–172, 736–797). **Verified the round-trip logic:**

**Good news — sharing DOES round-trip.** `writeThresholdsToURL()` (117) encodes only non-default thresholds as `?t.<critId>=<value>` via `replaceState`; `applyThresholdsFromURL()` (102) runs **before any render** at boot (line 1005, with a correct comment explaining why). So a copied filter URL restores slider/threshold state on load. The share button (736) flushes the debounce and copies `window.location.href`. This is correctly built and is a real viral loop. Credit where due.

**Weaknesses:**
1. The share button is **hidden until a filter is active** (`.match-bar .share { display: none }`, revealed by `has-filters`). Correct UX, but it means a visitor who hasn't touched a slider never sees that sharing exists — and with the modal locking sliders (C1), many never get to the state where Share appears.
2. The shared card preview won't reflect the filter (see H4) — every share shows the generic `og.png`.
3. "Share these criteria" copies a URL to clipboard with a toast, but there's no native share sheet (`navigator.share`) on mobile, where most social sharing happens. The clipboard fallback is solid, but a Web Share API path would be one tap to WhatsApp/X.

**Suggested fix:** Add `navigator.share()` on mobile with clipboard fallback; once C1 is fixed and sliders are usable, the share loop actually activates. **Effort: S.**

### M4 — `deeper.html` is under-linked and not in search index path
**Location:** linked from hero-cta (index 1167), contact invitation-cta (1268), footer (1297). Not in any sitemap (C2).

**What's wrong:** `deeper.html` is the strongest SEO asset on the property (long-form, sourced, keyword-rich: "ecovillage," "regenerative settlement," "permaculture," regional names, "where to settle"). It's reachable by humans but invisible to a sitemap and has no canonical. Its `og:image` is `og-deeper.png` (verified exists pattern). It deserves to be the search-traffic catcher.

**Suggested fix:** Sitemap entry (covered in C2); add JSON-LD `Article` structured data to `deeper.html` and `WebSite`/`Organization` to `index.html`; ensure internal links use descriptive anchor text (the hero-cta currently says "On the ethics of a framework like this" — good; the footer link "deeper material" is weaker). **Effort: M.**

## Top 3 time-sensitive opportunity moves (do these first)

1. **Unlock the sliders + soften the modal (C1).** Stop disabling the core interaction at 600ms and stop interrupting at a blind 3s timer. Let cold visitors experience the value (sliders + share loop) *before* asking for email. This is the biggest lever on both bounce and signup quality, and it's an S-effort change. Everything else compounds off it — the share loop (M3) and conversion only activate once people can actually use the page.

2. **Add measurement before the spike peaks (M2).** Fire a Vercel Analytics custom event on each signup success and on share-click. Without this you cannot tell whether any of these changes worked, and you only get this traffic once. One line per handler.

3. **Ship canonical + robots + sitemap, and de-"sketch" the title/OG (C2 + H2).** The spike generates backlinks; canonicalization and a sitemap capture that equity (especially for the strong `deeper.html`), and removing "sketch" from the `<title>`/share-card title stops underselling the work at the exact first-impression moment. All S-effort static-file/head edits.

**Honest caveat:** I have not load-tested anything, and I can't measure current bounce/conversion (no event tracking exists — that's finding M2). The conversion-risk assessments above are reasoned from UX patterns and the code's actual behavior, not from this site's data. The modal-gating critique (C1) is the most evidence-backed because it's a direct read of what the code does to a cold visitor; the "this will reduce signups" claim is a hypothesis worth A/B-confirming once M2 is in place.
